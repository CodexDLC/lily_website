# 06. ЛОГИКА ЗАПИСИ И TELEGRAM (BOOKING FLOW)

[📂 Design Index](./README.md) > 📄 Booking Flow

---

Этот документ описывает алгоритм работы кнопки "Записаться" и систему управления через мессенджер.

## 1. Сценарии Пользователя (Frontend Flow)

### Сценарий А: "Классическая запись" (Кнопка на Главной)
Клиент не знает, к кому хочет, ему нужна услуга.
1.  **Триггер:** Кнопка CTA на первом экране.
2.  **Шаг 1 (Категория):** Выбор: Парикмахерский зал / Ногтевой сервис.
3.  **Шаг 2 (Услуга):** Выбор конкретной процедуры (напр. "Стрижка Каре").
    * *Система:* Подтягивает `duration` (длительность) и цену.
4.  **Шаг 3 (Мастер):**
    * Опция А: "Любой свободный мастер" (максимум слотов).
    * Опция Б: Выбор конкретного специалиста из списка (тех, кто привязан к этой услуге).
5.  **Шаг 4 (Дата и Время):** Календарь показывает ТОЛЬКО те слоты, куда "влезает" выбранная услуга.
6.  **Финал:** Ввод Имени и Телефона -> СМС-код (опционально) -> "Спасибо".

### Сценарий Б: "Персональная запись" (Со страницы Мастера)
Клиент хочет именно к этому человеку.
1.  **Триггер:** Кнопка "Записаться к [Имя]" в профиле мастера.
2.  **Шаг 1:** Мастер уже зафиксирован.
3.  **Шаг 2:** Список услуг фильтруется (показываем только то, что делает этот мастер).
4.  **Далее:** Шаги 3-6 как в сценарии А.

## 2. Алгоритм "Тетрис Времени" (Backend Logic)
Система не предлагает жесткие слоты (10:00, 11:00). Она рассчитывает их динамически.

* **Входные данные:**
    * `Service Duration`: 90 мин.
    * `Master Schedule`: Свободен с 12:00 до 16:00.
* **Логика:**
    1.  Клиент выбирает услугу (90 мин).
    2.  Система сканирует график мастера.
    3.  Она находит "окна" >= 90 минут.
    4.  Выводит клиенту варианты старта: 12:00, 12:30, 13:00, 13:30 (позже нельзя, не успеет до 16:00).
* **Buffer Time:** Автоматически добавляем +15 минут после каждой услуги на уборку/отдых.

## 3. Telegram Admin (Управление)
Вместо сложной Админки на сайте, все управление происходит через Telegram-бота.

### Роли
* **Владелец (Superuser):** Видит всё, может отменять любые записи, менять график.
* **Мастер:** Видит только свои записи.

### Уведомления (Events)
1.  **Новая заявка:**
    * *Сообщение:* "🔥 Новая запись! Клиент: Анна (+49...). Услуга: Стрижка. Время: Завтра 14:00."
    * *Кнопки:* [Подтвердить] | [Отклонить] | [Связаться].
2.  **Напоминание (Утро):**
    * "Доброе утро! Сегодня у вас 4 клиента. Первый в 10:00."

---

## 4. Waitlist (Лист Ожидания)
*Если свободных мест нет:*
1.  Клиент видит сообщение: "К сожалению, на эту дату мест нет".
2.  Появляется кнопка: **"Сообщить, если освободится место"**.
3.  Клиент оставляет телефон.
4.  Если кто-то отменяет запись -> Администратору приходит алерт: "Освободилось окно! В листе ожидания есть 2 человека, позвоните им".

## 5. Технические требования к БД
Для реализации этой логики модель данных (`models.py`) должна строго содержать:
* `Service.duration_minutes` (Integer).
* `Specialist.telegram_id` (Для отправки уведомлений).
* `Booking.status` (New / Confirmed / Cancelled).
