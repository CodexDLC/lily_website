# 📬 Модуль Уведомлений (Notifications)

[⬅️ Назад в корень](../../README.md)

Подсистема `notifications` спроектирована так, чтобы полностью изолировать бизнес-логику создания писем и сообщений от транспортного слоя их отправки. Она обрабатывает форматирование и сборку исходящих сообщений (Emails, SMS, Telegram) независимо от фреймворка.

## 🏗️ Архитектурная концепция

Исторически отправка email-ов в Django зашита глубоко в сервисы, что делает код нетестируемым и медленным (синхронные вызовы `send_mail`).

В `codex_tools`:
1. **Ядро формирует только пакет данных (Builder)**. Вместо рендеринга HTML или подключения к SMTP, `NotificationPayloadBuilder` просто создает стандартизованный словарь (JSON) с переменными (`context_data`) и списком нужных каналов (`channels`).
2. **Ядро передает пакет Адаптеру**. Встроенный `BaseNotificationEngine` принимает готовый словарь и говорит Адаптеру: "Положи это в очередь".
3. **Отправка происходит снаружи**. На стороне инфраструктуры запущен отдельный воркер (ARQ или Celery), который берет этот JSON, сам находит нужный HTML-шаблон по `template_name`, рендерит его и отправляет через внешний API (Postmark, Twilio, Telegram).

Это обеспечивает идеальное разделение ответственности (Separation of Concerns).

## 🗂️ Карта Модуля

### `NotificationPayloadBuilder` (`builder.py`)
Централизует создание словаря полезной нагрузки (payload). Он гарантирует, что любая отправленная задача будет иметь правильно отформатированные ключи:
- `notification_id`
- `recipient` (email, телефон, имя)
- `template_name`
- `context_data` (переменные для рендеринга шаблона)
- `channels` (список разрешенных каналов отправки, например `['email', 'telegram']`)

### `BaseNotificationEngine` (`service.py`)
Класс-оркестратор. Модули, использующие `codex_tools`, должны инстанцировать его (или наследоваться от него), передав `adapter`, который реализует метод `.enqueue(task_name, payload)`.

### `Адаптеры (Adapters)` (`adapters/`)
Как и в модуле бронирования, связь с фактическим фреймворком проекта происходит через адаптеры. Например, `django_adapter.py` оборачивает клиенты ARQ или Celery, чтобы отправлять словари, созданные `Builder`, в реальные очереди Redis.
