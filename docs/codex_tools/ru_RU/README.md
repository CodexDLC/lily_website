# 🛠️ Codex Tools (Архитектура)

[⬅️ Назад к выбору языка](../README.md)

Папка `codex_tools` изначально проектировалась с прицелом на вынесение в **отдельную корпоративную библиотеку (PyPI-пакет)**. В ней собраны ключевые бизнес-алгоритмы, которые должны быть универсальны для разных проектов.

## 🏛️ Гексагональная архитектура (Порты и Адаптеры)

Главная архитектурная особенность `codex_tools` — это **нулевая зависимость от фреймворка** (в частности, от Django) в "ядре" логики.
Мы используем паттерн "Порты и Адаптеры":

1. **Доменное Ядро (Domain / Engine)**:
   Файлы вроде `slot_calculator.py` или `chain_finder.py`. Это чистый Python. Ядро ничего не знает о SQL, базе данных, HTTP-запросах или ORM.
2. **Порты (входящие / исходящие)**:
   Всё общение с внешним миром строится на:
   - **DTO (`dto.py`)**: Строгие, неизменяемые модели Pydantic (иммутабельность защищает от случайных побочных эффектов).
   - **Интерфейсы (`interfaces.py`)**: Протоколы `typing.Protocol`, описывающие контракты того, что системе нужно от внешнего мира (например, контракт `ScheduleProvider`).
3. **Адаптеры (`adapters/`)**:
   Это единственные "грязные" зоны, где разрешены импорты из фреймворка (Django). Адаптер читает модели в базе данных и "перекладывает" их в чистые Pydantic-модели (DTO) для ядра.

### Зачем нужен такой подход?

- **Отчуждаемость**: Папку можно просто перенести в проект на FastAPI и SQLAlchemy. Потребуется написать только новый адаптер в пустую папку `adapters/`.
- **Тестируемость**: Логика расчёта слотов легко покрывается юнит-тестами без необходимости поднимать базу данных (`tests/*.py`).

## 🗺️ Ключевые слои библиотеки

### Подсистема бронирования (`booking/`)
Полностью независимый от базы алгоритм для расчета записей к исполнителям с учетом:
- Многосоставных/комбо услуг.
- Динамических окон доступности.
- Настроек перерывов и "гэпов" после процедуры.

### Подсистема нотификаций (`notifications/`)
Позволяет собирать сложные бизнес-уведомления (Email, SMS) через интерфейсы и абстрактные селекторы шаблонов, чтобы бизнес-логике не приходилось заниматься рендерингом строк.

### Пул общих утилит (`common/`)
Набор инструментов, постоянно мигрирующих между проектами: стандартизованный логгер на базе `loguru` (перехватывающий стандартный питоновский `logging`), утилиты для кеширования Redis и клиент фоновых задач (например `ARQ`).
